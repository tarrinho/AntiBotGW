## Features validated in 1.7.7

### GW Mgmt filter fix
`protect()` middleware now calls `record()` for authenticated operator requests before returning. Previously returned early without recording → gwmgmt filter showed zero entries.

### Geo dashboard loading/ready pill
Added `#load-status` CSS pill to `dashboards/geo.html`:
- Starts yellow with pulsing dot animation ("Loading") on page open
- Flips to solid green "Loading Ready" inside double `requestAnimationFrame` after first successful `tick()` data fetch
- Matches the controls dashboard pattern exactly

### Dashboard data test fixes
Three stale tests in `test_dashboard_data.py` updated to match current endpoint response shapes:
- `agents-data` key: `agents` → `suspects`
- `logs-data` key: `events` → `rows`
- `path-hits`: added required `?path=/` param; key `paths` → `ips`

### Dashboard chart bug fix
`main.html` tick() timeline block used `'blocked'` as the filter key for `datasets[2].hidden` — corrected to `'ban'` / `'reallyban'` so blocked line no longer hidden unconditionally.

### BYPASS_PATHS audit trail
`protect()` bypass early-return now writes `("event", ..., "bypass-path")` to `db_queue` before returning. Every bypassed access is traceable in the events log and logs dashboard. `ip_state` intentionally stays empty (no bot scoring overhead).

### Main dashboard path search + event log panel
- Path search input in the filter bar filters clients table by `last_path` substring match
- "Path event log" panel (hidden when no query) fetches `/secured/logs-data?q=<path>` and renders all matching DB events including `bypass-path` entries (time, IP, path, status, reason, UA)
- Debounced 300 ms; clear button

### MaxMind in-process lookup cache
`reputation/maxmind.py` — Added `_asn_cache` / `_city_cache` dicts with 24-hour TTL and 8 192-entry FIFO eviction. Eliminates repeated mmdb disk reads (previously ~4 reads/request for same IP: asn×2, city×2). Cache check in `_city_lookup` moved before the reader-null guard so cached results survive monthly mmdb refresh cycles (without this ordering, a refresh that sets `_city_reader = None` momentarily would bypass the cache and return None).

### Service metrics 30-day retention window
`config.py` — `SERVICE_METRICS_INTERVAL` default changed 10 s → 60 s; `SERVICE_METRICS_RETENTION` default 4 320 → 43 200. At 60 s/sample the system now retains 30 days of service data at ≈ 22 MB in-memory. Both values remain overridable via `SVC_METRICS_INTERVAL` / `SVC_METRICS_RETENTION` env vars.

### `controls.html` Apply/Reset button placement
`dashboards/controls.html` — "Apply changes" / "Reset" action bar moved from below the Thresholds card to immediately before the Defenses & Scoring section. Buttons are now visible without scrolling past the entire form.

### `controls.html` `url()` identity function restored
`dashboards/controls.html` — DC-01 cleanup in a previous iteration incorrectly removed `const url = p => p` from controls.html, which has ~30 `url(path)` call-sites in fetch() chains. Restored. Without it all dashboard panels (Lists, Bans, External integrations, Thresholds, etc.) threw `ReferenceError: url is not defined` on load.

### `logs.html` category filter pills
`dashboards/logs.html` — Five toggle pills (Allowed · Ban · Really Ban · Auth Bots · GW Mgmt) appear in a filter bar when the Requests tab is active. `_logCat(r)` classifies each row: `authorized-robot` reason → authbots; path starting `/antibot-appsec-gateway/` → gwmgmt; hard-ban reasons (canary-echo / honeypot-silent / honeypot) → reallyban; any non-OK reason → ban; else → allowed. Filtering is client-side (no network round-trip); row count updates live.

### `geo.html` ready-state pill text corrected
`dashboards/geo.html` — Ready-state pill text was "Ready"; corrected to "Loading Ready" to match the JS flip logic intent and validation spec.

### Geo-map 30-day view
`dashboards/geo.html` — added `30 days` option (value 43200) to the window `<select>`. `core/proxy_handler.py` — raised range cap in `geo_data_endpoint` and `geo_drill_endpoint` from 10 080 → 43 200 minutes. Events table is never pruned so data depth matches retention. Two performance countermeasures applied for large windows:
- **Cursor iteration** (no `fetchall`) — rows stream from SQLite one batch at a time; constant RAM regardless of window size.
- **Reservoir sampling (Algorithm R)** — `events_sample` (5 000-slot scrubber buffer) is now a uniform random sample across the full window. Previously the buffer was filled with the first 5 000 chronological events (all from the oldest slice), making the scrubber replay non-representative for windows > a few hours.
`ORDER BY ts ASC` removed from the query — not needed because `rebuildBuckets()` bins each event by its `ts` value directly.

### `rules.md` step 14e — orphan image cleanup
Post-push step added: `docker image prune -f` after all three arch images are pushed to Harbor, removing dangling `<none>` tags left by the build process.
