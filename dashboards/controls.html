▶ Activation order — when to enable each tier
11st Order — Baseline
Always active. Zero-config, microsecond cost, near-zero false-positive risk. Catches obvious bots: bad/empty UA, basic flood, canary traps, header fingerprints. Safe to enable on day 1 before any traffic analysis.
Enable when: deploying the gateway — no prior evidence needed.
22nd Order — Elevated Suspicion
Enable when bot patterns appear in logs. Behavioral and AI-specific signals, regex body scans, IP reputation basics (Tor, ASN). May occasionally flag API clients — review false positives before locking in.
Enable when: logs show suspicious patterns OR risk_score frequently hits 15–30 range.
33rd Order — Confirmed Attack
Aggressive or expensive controls: external API calls (AbuseIPDB, CrowdSec), geo/DC blocking, JS challenges, tarpits. Require configuration and have latency cost or collateral damage risk. Use when 1st+2nd order is insufficient.
Enable when: active attack confirmed OR escalation_threshold frequently reached OR 1st+2nd insufficient.
