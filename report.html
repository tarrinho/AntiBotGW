1. Executive Summary

AppSecGW is a production-grade reverse HTTP proxy implementing layered anti-automation and web application firewall capabilities. It sits in front of any upstream HTTP service and provides real-time threat detection, risk scoring, automated banning, operator dashboards, and security event visibility — all within a single ~79 MB distroless container with zero runtime CVEs.

Release 1.8.4 adds the SIEM Security Event Center, a new operator dashboard providing a unified threat-intelligence view across all detectors; implements M-SEC-1 upstream address scrub to prevent internal infrastructure disclosure in proxied responses; enforces STRICT_VHOST mode by default; and fixes two silent url() TypeError bugs in the Live Feed dashboard plus a Chart.js date-adapter crash in the Traffic by Virtual Host chart.

2.2 Request lifecycle

Inbound request │ ├─ STRICT_VHOST check (v1.8.4) ─── unknown host → 502 decoy │ ├─ Ban list check ──────────────── banned IP → 404 silent-decoy │ ├─ Detector pipeline (sequential) │ UA blocklist · suspicious-path · body-sqli / body-cmd │ body-ssrf / body-rce / body-xss · DLP · rate-limit │ JS challenge · fingerprint / JA4 · AI-agent probes │ Risk score accumulated per-request │ ├─ Auto-ban if risk ≥ threshold ─── ban written to ip_state + DB │ ├─ proxy_handler() ─────────────── forward to upstream │ └─ M-SEC-1 response scrub (v1.8.4) strip upstream addr → client

2.3 State model

All per-IP state is held in a defaultdict[str, IpState] guarded by asyncio.Lock. Events are persisted asynchronously to SQLite (WAL mode) for dashboard queries without blocking the request path. Fleet mode (optional): state shared across multiple instances via Redis.

3. Detection Layers

Detectors run in sequence inside protect(). Each detector adds a risk score on signal fire. When the cumulative score for an IP reaches RISK_BAN_THRESHOLD (default 50) the IP is banned automatically.

# Detector / signal Default score Description
1honeypot / honeypot-silent+100 Hard-coded trap paths (/wp-admin, /.env, etc.) and HTML-injected canary links. Instant ban on first hit.
2ua-blocked+60 Substring match against automation fingerprints (curl, python-requests, scrapy, selenium, headless, claude, gptbot, …).
3suspicious-path+40 Regex match for injection patterns in URL path: XSS (<script>), path traversal (../), SQL (UNION SELECT), SSTI ({{7*7}}).
4body-sqli / body-cmd / body-rce / body-ssrf / body-xss+50 Body injection scan (threshold-gated: only fires when risk ≥ ESCALATION_THRESHOLD). Regex-based detection on request bodies for SQL, OS command, JNDI/log4shell, SSRF, and XSS payloads.
5rate-limitban Per-IP token bucket. Configurable burst and refill rate. Burst exhaustion → 429 with Retry-After.
6banned-silent— IP is in the ban list. Served a 404 silent decoy; no signal emitted to attacker about ban status.
7js-challengeconditional JS proof-of-work / Cloudflare Turnstile challenge gate (configurable). Clients that fail or skip receive 403.
8fingerprint / ja4+30–60 TLS JA4 fingerprint blocklist and consistency checks.
9AI-agent probes (v1.7.3)+35–90 Honey credentials, redirect maze, LLM no-subresource heuristic, browser canary probe, path sweep.
10DLP (optional)alert Outbound data loss prevention: AWS key patterns, credential strings in response bodies → webhook alert + redaction.

4.2 M-SEC-1 — Upstream Address Scrub

Every proxied response is now passed through an unconditional scrub step that removes all references to the upstream address before forwarding to the client. This prevents internal infrastructure disclosure (scheme, hostname, port) even when the upstream leaks its own address in response headers or bodies.

Header handling

Header typeAction
Location, Content-Location, Link, Refresh, Access-Control-Allow-OriginUpstream origin replaced with gateway origin
Via, Server, X-Powered-By, X-Backend, X-Upstream, X-Origin, X-Real-Server, X-Forwarded-ServerDropped if they contain the upstream address
All other headers containing upstream addressScrubbed (value replaced)

Body handling

Content-TypeAction
text/*, application/json, application/xml, application/javascript, and similar text typesUpstream scheme://netloc replaced with gateway origin in body text
Binary types (image/*, audio/*, application/octet-stream, etc.)Passed through untouched

Activated unconditionally. No env var opt-out (by design — information disclosure is always undesirable). Gateway origin derived from the inbound Host header + scheme.

4.3 STRICT_VHOST — Default ON

When at least one virtual host is registered, STRICT_VHOST=1 (the new default) rejects inbound requests whose Host header does not match any configured vhost. Unmatched hosts receive a 502 decoy with no information disclosure. Guard condition: 

if STRICT_VHOST and VHOSTS and not vhost_is_configured(host):
    return web.Response(status=502)  # silent decoy

Has no effect in single-site deployments where VHOSTS is empty. Set STRICT_VHOST=0 to allow unknown hosts to fall through to the global UPSTREAM.

4.4 Bug Fixes

Live Feed — Detection Methods / Top Methods panels always empty

loadDetectorStats() called url('/antibot-appsec-gateway/secured/detector-stats') where url is not a function in that scope. The resulting TypeError was silently swallowed by a bare catch(e){}, leaving both panels perpetually empty. The same pattern affected loadLogLevel(). Both fixed to use bare string paths.

Traffic by Virtual Host chart crash

Chart.js type:'time' axis requires a registered date adapter (date-fns or luxon); none is bundled with the local chart.umd.min.js. The runtime error "This method is not implemented: Check that a complete date adapter is provided" crashed the chart on every render. Fixed by switching to type:'category' with pre-formatted fmtTime() string labels — identical to the main traffic chart approach.

Traffic by Virtual Host — Click-to-inspect

Clicking any bar in the vhost stacked-area chart pins a bucket detail panel below the canvas showing all vhosts for that time bucket, sorted by request count with an inline share percentage bar. Clicking the same bar a second time unpins. The panel auto-updates on each poll (pinned bucket index clamped to new data length). State stored in module-level _vhRawData and _vhSelectedIdx.

4.5 UI/UX Fixes

Dead nav links — center_control.html

Sidebar "Center Control" link pointed to /secured/center-control (non-existent route). "Dashboard" link pointed to /secured/dashboard (removed in 1.7.x). Fixed: "Center Control" → /secured/control-center; "Dashboard" replaced by "Live Feed" → /secured/live-feed.

Silent error swallowing — main.html

_attackerBan and _attackerUnban action handlers had .catch(function(){}) — failed bans/unbans were silently discarded with no operator feedback. Fixed: catch now calls _gwAlert('Ban/Unban failed: ' + message).

Duplicate API calls — control_center.html

DOMContentLoaded called loadSignalPerf() and loadThreatDonut() directly, plus they were called inside _loadThreatSection() — every page load issued two concurrent fetches to each endpoint. Fixed: removed the redundant direct calls from DOMContentLoaded; _loadThreatSection() is the sole entry point.

Sidebar nav sub-items (all 11 dashboard pages)

Live Feed, Agents, and SIEM now appear as indented sub-items under Control Center in every dashboard sidebar, using class="sub" (padding-left:20px; font-size:11.5px). SIEM moved from end-of-nav to immediately after Agents. Active page retains class="sub active".

ARIA-live toast notifications (all pages with toast)

All <div id="toast"> elements now carry role="status" aria-live="polite" aria-atomic="true" so screen-reader users hear action confirmations (ban, unban, config save, etc.).
