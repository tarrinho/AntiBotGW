1. Executive Summary

AntiBot/WAF GW is a reverse HTTP/HTTPS proxy that sits in front of a protected upstream application and filters automated and malicious traffic before it reaches the origin. Version 1.9.9 extends the original six-layer anti-automation design into a full layered defense stack combining bot/agent detection, a request-time Web Application Firewall (WAF), IP / reputation intelligence, and a risk-scoring engine with an operator console of eleven dashboards.

Every request passes through an aiohttp middleware chain. Each layer addresses a distinct attack surface used by CLI tools, scrapers, headless browsers, AI agents, and exploitation frameworks. Detections feed a weighted risk score; once a score crosses the (NAT-aware) ban threshold the identity enters a hostile pool and is served a silent decoy — the upstream homepage or a 404 — so the client cannot tell it was caught.

Implemented controls (1.9.9)

  • 36 weighted detection signals across UA, path, header, behavioural, fingerprint, and reputation families, plus 13 hot-toggleable kill-switches and a decay-aware risk model with NAT-aware thresholds and an optional strict-PoW "Anubis" mode.
  • Request-time WAF: HTTP request-smuggling, verb-override, header injection (SSTI / host-header), body inspection (XXE, prototype-pollution, critical injection), file-upload, GraphQL abuse, and slow-loris — each behind its own WAF_*_ENABLED kill-switch.
  • Reputation & threat intel: AbuseIPDB, CrowdSec, MaxMind geo/ASN, Tor exit list, and the Feodo / CINS / URLhaus C2 + malware feeds.
  • Challenge stack: heuristic JS cookie gate, optional Cloudflare Turnstile, proof-of-work, and an AI-labyrinth / tarpit for crawlers that follow hidden links.
  • JA4 TLS-fingerprint binding of the challenge cookie when a JA4-injecting front (cloudflared) is present.
  • Ed25519 gateway mesh: peer gateways exchange signed reputation offers; only offers verifying against a registered peer public key are applied.
  • 196 hot-reload knobs, all tunable at runtime via /secured/config and the operator dashboards without a restart.
  • Dual datastore (SQLite default, optional PostgreSQL/TimescaleDB with dual-write) for bans, config, audit, and observational metrics that survive restarts and session-key rotation.

The build ships as a multi-arch container image (amd64 / arm64 / arm/v7) with a clean CVE posture (0 CRITICAL / 0 HIGH on all three arches) and is validated by a 20-stage release pipeline covering unit/functional/integration tests, mutation testing, SAST, dependency/image CVE scans, dynamic security testing, and a written per-release audit record.

2. Architecture

2.1 Deployment topology

The gateway runs as a single container behind a TLS-terminating front (Cloudflare Tunnel in the reference deployment, which also injects the client CF-JA4 TLS fingerprint). The front forwards to the gateway's listener; the gateway applies all detection + WAF layers and proxies clean traffic to the upstream origin.

Client │ HTTPS ▼ ┌─────────────────────────────────┐ │ TLS front (Cloudflare Tunnel) │ ← terminates TLS, injects CF-JA4 │ https://app.example.com │ └───────────────┬─────────────────┘ │ HTTP + X-Forwarded-*, CF-JA4 ▼ ┌─────────────────────────────────┐ │ AntiBot/WAF GW (this build) │ │ protect() middleware │ │ ├─ trust topology / IP-ban │ │ ├─ detection layers (L1..L9) │ │ ├─ WAF layers (L10) │ │ ├─ risk score → ban / decoy │ │ └─ proxy clean → upstream │ └───────────────┬─────────────────┘ │ HTTP ▼ ┌─────────────────────────────────┐ │ Upstream origin (app.example) │ └─────────────────────────────────┘

2.2 Request lifecycle

Every request flows through the protect() middleware. Short-circuit checks (IP-ban, admin-path gating, bypass paths, authorized-bot pass-through) run first; then the detection + WAF layers accumulate risk; finally the risk verdict selects forward / challenge / silent-decoy / ban.

Request → protect() middleware 1. Source-IP resolution (XFF trust topology) + persistent IP-ban check 2. Admin-path gating (admin-IP allowlist + session) / authorized-bot rules 3. Detection layers L1..L9 (accumulate weighted risk) 4. WAF layers L10 (smuggling / verb / header / body / upload / graphql / slowloris) 5. Challenge gate (JS cookie / PoW / Turnstile) when required 6. Risk verdict: score < SOFT_CHALLENGE_SCORE → forward to upstream soft band → monitor / challenge score ≥ RISK_BAN_THRESHOLD → hostile pool + silent decoy 7. Rate-limit (per-IP + per-identity token buckets, global RPS cap) 8. Forward clean traffic → upstream; record event → datastore

2.3 State model

Hot-path state (per-identity risk, token buckets, session/fingerprint maps, recent events, timeline buckets) lives in memory for zero-latency reads. Durable state (bans, config_kv, secrets, admin-IP ACL, audit, observational metrics) is persisted to the datastore and rehydrated on boot. Security-critical ban state is rehydrated synchronously before the server accepts traffic; cosmetic dashboard state is rehydrated in a background task so the gateway accepts connections within seconds of a restart. The persistent ip_bans table keys on the raw IP and therefore survives session-cookie and fingerprint rotation.

3. Protection Layers

The layers below are grouped by attack surface. Each weighted signal is independently hot-toggleable and contributes to the identity's risk score; WAF layers are terminal (immediate silent-decoy on a confirmed match).

1User-Agent & client identity

Blocklist of 60+ UA fragments (HTTP libraries, scanners, headless browsers, and AI crawler bots such as gptbot, claudebot, anthropic), plus empty / too-short / non-browser / platform-mismatch signals against the declared Sec-CH-UA-Platform.

2Path & honeypot traps

Suspicious-path regex (VCS metadata, backup files, traversal, SQLi/XSS/LFI markers), advertised honeypot paths with no upstream (/.git/HEAD, /.env, /actuator/env, …), hidden bot-trap form fields, and an AI-labyrinth that bans clients which follow injected rel=nofollow links.

3Header completeness & consistency

Required-header allowlist, Accept-* / Sec-Fetch-* presence, Sec-Fetch navigation coherence, Client-Hints vs UA consistency, and an HTTP/2 SETTINGS fingerprint check.

4Behavioural analysis

Inter-arrival timing regularity, path-enumeration thresholds, HTML-without- assets ratio, and request-cadence autocorrelation — robotic-pattern signatures.

5Challenge & proof-of-work

Heuristic JS cookie gate (auto-mint on first qualifying HTML GET), optional Cloudflare Turnstile, and a proof-of-work gate (strict "Anubis" mode for configured paths). Unsolved challenges are silently decoyed.

6Rate limiting

Per-IP and per-identity token buckets plus a global RPS cap; throttled traffic receives HTTP 429 with no risk penalty (pure back-pressure).

7Canary & referer integrity

Canary tokens planted in responses (echo detection for LLM agents that replay page content) and referer-chain validation (ghost / loop signals).

8TLS / JA4 fingerprint

JA3/JA4 handshake fingerprints matched against a deny-list; the challenge cookie is bound to the observed JA4 so it cannot be replayed under a different TLS stack.

9Reputation & threat intelligence

AbuseIPDB confidence tiers, CrowdSec decisions, MaxMind geo/ASN (country allow/deny + hosting-ASN signal), Tor exit detection, and the Feodo / CINS / URLhaus C2 + malware feeds.

10Web Application Firewall

Request-time WAF inspecting method, headers, and (bounded) body: HTTP request-smuggling, verb-override, header injection (SSTI / host-header), XXE, prototype-pollution, critical command/SQL injection, malicious file-upload, GraphQL introspection/batching abuse, and slow-loris. Each detector is gated by its own WAF_*_ENABLED kill-switch (default-on, per-vhost overridable) and yields a silent decoy on a confirmed match.

Cross-cutting: a decay-aware weighted risk model (SOFT_CHALLENGE_SCORE / RISK_BAN_THRESHOLD / RISK_BAN_THRESHOLD_NAT / RISK_DECAY_HALFLIFE_SECS / HOSTILE_BAN_SECS); silent-decoy responses (BLOCK_RESPONSE_MODE = homepage | 404); a persistent raw-IP hostile pool; and an Ed25519-signed gateway mesh for sharing reputation between peers.

Admin endpoints are gated by an admin-IP allowlist plus an authenticated session; unauthenticated requests receive an upstream-style 404 (the route's existence is never confirmed). 6. Bypass Surface & Residual Risk

The accepted-risk register (re-confirmed each release in validation/<version>.md §11b) records design trade-offs that are not defects but carry residual exposure:

IDResidual riskMitigation
R1agw_csrf cookie is JS-readable (HttpOnly off)Double-submit token + SameSite=Strict
R2ALLOW_PRIVATE_UPSTREAM default-allow (SSRF reach)Operator-scoped; SSRF guard on outbound URLs; env-pinnable
R3Admin surface hidden behind a decoy-404Admin-IP allowlist is the real gate (obscurity is a layer, not the control)
Front-of-proxy trust boundary: the gateway's source-IP and JA4 guarantees depend on the front. It must terminate TLS, inject a trustworthy JA4, and overwrite (not append) the client-supplied X-Forwarded-For. A front that passes through client-controlled XFF undermines per-IP bans and geo/reputation lookups; pin the trusted front via TRUST_XFF + JA4_TRUSTED_PEERS.

7. Recommendations

  1. Deploy behind a JA4-injecting, XFF-overwriting front (Cloudflare Tunnel or an nginx JA4 build) and pin it with TRUST_XFF=last + JA4_TRUSTED_PEERS.
  2. Keep the WAF kill-switches default-on; disable a layer only per-vhost when a legitimate client trips it, and re-enable after tuning.
  3. Enable PostgreSQL/TimescaleDB for multi-replica or long-retention deployments; SQLite is sufficient for single-instance.
  4. Configure reputation feeds (AbuseIPDB / CrowdSec) and the C2/malware feeds for environments exposed to broad internet scanning.
  5. Set LOG_RETENTION / DLP redaction in line with the privacy posture — IP addresses are PII; prune the events store on schedule.
  6. For mesh deployments, upgrade all peers together (the Ed25519 model is not compatible with the legacy symmetric keys) and rotate keypairs.

8.3 Verdict

AntiBot/WAF GW 1.9.9 implements a defense-in-depth stack — bot/agent detection, a request-time WAF, reputation intelligence, and a decay-aware risk model — with a clean CVE posture and a complete, version-controlled validation record. The dominant residual risk is the trust boundary at the front of the proxy; the documented deployment guidance closes it.
